CentOS启用APR+Https

发表于 2017-09-07

0x00 背景

  最近我司一项目tomcat需要全站Https，客户提供了2个文件：

  • xxx.crt
  • xxx.key

  系统运行环境：

  • CentOS 6.x
  • Tomcat 7.x
  • JAVA7

0x01 tomcat运行模式

  对于一般个人用户来说，tomcat启用Https可以通过jvm提供的keytool工具来生成.keystore文件，然后在tomcat中配置keystoreFile和keystorePass，keytool位于/path/to/JDK/jre/lib/security/目录。但是自己生成的证书文件，浏览器是不认可的。所以，自己生产证书文件并配置的Https在浏览器中会显示站点连接不可信，并有红色反斜杠标志，没有我们希望得到的保护色的漂亮Https标志。

  由于我们客户是提供了证书文件，而且用户希望看到保护色的漂亮标志，因此，我们必须利用客户提供的文件来启用全站Https。

  tomcat有3种运行模式，如下：

运行模式 protocol 说明
BIO org.apache.coyote.http11.Http11Protocol 阻塞式I/O，tomcat8一下默认启用模式。
NIO org.apache.coyote.http11.Http11NioProtocol 非阻塞式I/O，通过JDK提供的java.nio.*实现，tomcat8默认启用模式。
APR org.apache.coyote.http11.Http11AprProtocol 非阻塞式I/O，通过系统级类库APR实现，性能最高，默认不启用。

APR：Apache Portable Run-time libraries，Apache可移植运行库
在早期的Apache版本中，应用程序本身必须能够处理各种具体操作系统平台的细节，并针对不同的平台调用不同的处理函数。随着Apache的进一步开发，Apache组织决定将这些通用的函数独立出来并发展成为一个新的项目。这样，APR的开发就从Apache中独立出来，Apache仅仅是使用APR而已。

  在这三种模式中，Https的配置还不一样，BIO和NIO模式使用keystoreFile和keystorePass来配置，而APR模式中使用SSLCertificateFile和SSLCertificateKeyFile来配置。由于客户提供给我们的是.crt文件，而不是.keystore文件，因此我们只能使用APR模式。

java中的动态代理

java中的动态代理主要有3种：

  • AspectJ
  • JDK动态代理
  • CGLIB

AspectJ通过在编译时织入代码来增强接口或类的功能，不属于java的语法

三种代理区别：

动态代理 代理模式
AspectJ 编译时增强
JDK 运行时增强，主要面向接口
CGLIB 运行时增强，主要面向类

SpringAOP中的代理：

动态代理 配置方式 说明
AspectJ <aop:aspectj-autoproxy/> SpringAOP中的AspectJ只是利用JDK动态代理对AspectJ的一种实现，并没有使用AspectJ，动态代理对象必须实现一个或多个接口。
CGLIB <aop:aspectj-autoproxy proxy-target-class="true"/> 使用CGLIB作为动态代理时，会生成$$EnhancerByCGLIB$$这样的类。

项目中配置了2个context，而且这两个context属于包含与被包含的关系，通过分析tomcat的valve机制，找到问题原因在于redis中取出session进行反序列化操作的时候反序列化到/uploadfile这个context中，而这个/uploadfile中是没有SimplePrincipalCollection这个类的，所以报了java.lang.ClassNotFoundException。

编译安装nginx+sticky

发表于 2017-04-28

0x00 关于

sticky模块（nginx-sticky-module）
普通安装的nginx是不带sticky模块的，安装sticky模块需要重新编译nginx。
没有sticky模块的nginx主要通过ip_hash把同一来源的客户(同一C段的IP)固定指向后端的同一台机器，这样并不能实现很好的负载均衡，（针对同一网段的用户）。
增加了nginx-sticky-module的nginx，首先根据轮询RR随机到某台后端，然后在响应的Set-Cookie上加上route=md5(upstream)字段，第二次请求再处理的时候，发现有route字段，直接导向原来的那个节点。
如下:
route=e6b57a7d2c523d67adf934eb8213f4c8; JSESSIONID=ABE2BF19945483F6EB73DB7591F4838D

0x01 配置说明

  • name: 可以为任何的string字符,默认是route
  • domain：哪些域名下可以使用这个cookie
  • path：哪些路径对启用sticky,例如path/test,那么只有test这个目录才会使用sticky做负载均衡
  • expires：cookie过期时间，默认浏览器关闭就过期，也就是会话方式。
  • no_fallbackup：如果设置了这个，cookie对应的服务器宕机了，那么将会返回502（bad gateway 或者 proxy error），建议不启用。
